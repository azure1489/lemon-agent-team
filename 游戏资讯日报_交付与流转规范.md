# 游戏资讯日报 · 交付与流转规范

> 适用范围：本任务（微信公众号「游戏资讯日报」）下所有 Agent 的交付物。
> 核心：每个 Agent 的交付物 = 一个「`index.md` 主文档 + 附件」的文件夹，打包成 zip 上传 OSS，Agent 之间传**链接**流转，文件名带**责任方**、`index.md` 头记**溯源**。
> 中枢：**主编**是唯一对接用户的人，每一步都「审核 → 退回 / 转用户 → 用户通过后重新打包写意见与下一步安排再派工」，**主编自己不编辑内容**。

---

## 0. 角色与职责

| 角色 | 职责 | 不做 |
|---|---|---|
| **主编** | 审核每一阶段是否达验收标准；不过则退回并告知改的方向和位置；通过则转用户审；用户通过后重新打包、写上自己的意见和下一步安排，派给下一个 Agent；是唯一对接用户的人 | 不具体编辑修改内容 |
| **游戏情报员** | 在 X、Reddit 采集游戏相关资讯，产出资讯包 | — |
| **选题研究员** | 综合资讯包，输出标题 + 摘要 | — |
| **视觉设计师** | 据封面要求做封面图，产出图片 + 意见 | — |
| **深度内容作者** | 据选题 + 情报输出 md 格式文章及配图 | — |
| **发布员** | 输出 HTML，上传公众号草稿箱，先通知主编 | — |

## 1. 交付形态（文件夹）

交付物是一个文件夹，内含：
- `index.md`：固定入口主文档，正文内容直接内联在里面。
- 附件：图片、HTML、资讯原文等其他文件，放同目录（可按类型分子目录，如 `images/`、`source/`、`html/`）。
- `index.md` 用相对链接引用附件，如 `![](images/封面.jpg)`。

```
游戏资讯日报_深度内容作者_20260604_v1/      ← 打包成同名 .zip
├── index.md                               ← 固定入口
└── images/01-正文配图.jpg
```

## 2. 打包、上传与流转

1. 文件夹打包成同名 zip。
2. zip 上传 OSS，返回一个**完整的 https 链接**。
3. 链接是 Agent 之间流转的东西：下游 Agent 拿到链接 → 下载 zip → 解压 → 以 `index.md` 为入口读正文、按相对链接看附件。
4. 每个 zip 留在 OSS 作留底。

## 3. 命名规则

格式：`用途_责任方_日期_版本.zip`，本任务 `用途` 固定为 **游戏资讯日报**。
- 责任方：单一 Agent 产出写 **Agent 名**；多方合成产出写**阶段名**（成品）。
- **主编的派工单**也算单一产出，责任方写 **主编**，类型写「派工单」。
- 退回重做递增版本号（v1 / v2 / v3…），每版各打一个 zip、各自留底。

示例：
- `游戏资讯日报_游戏情报员_20260604_v1.zip`
- `游戏资讯日报_主编_20260604_v1.zip`（派工单）
- `游戏资讯日报_成品_20260604_v1.zip`（视觉 + 内容合成）

## 4. OSS 路径（分阶段归档）

`csw/{用途}/{日期}/{阶段序号-阶段}/{zip}`，例如：

```
csw/游戏资讯日报/2026-06-04/02-选题/游戏资讯日报_选题研究员_20260604_v1.zip
```

阶段序号：**01-情报 / 02-选题 / 03-视觉 / 04-内容 / 05-成品 / 06-发布**。**路径管排序，文件名管识别。**
主编派给某阶段的派工单，归到**该目标阶段**路径下，与该阶段的 Agent 产出并列（靠责任方区分），例如：

```
csw/游戏资讯日报/2026-06-04/02-选题/游戏资讯日报_主编_20260604_v1.zip      ← 主编派工单
csw/游戏资讯日报/2026-06-04/02-选题/游戏资讯日报_选题研究员_20260604_v1.zip  ← 选题研究员产出
```

> **链接 = OSS 域名 + 上面的路径。** Agent 之间流转、写进 `index.md` 的 `上游来源` 里的，必须是 OSS 返回的**完整 https 链接**（不能只写路径），例如：
>
> ```
> https://csw-oss.oss-cn-hangzhou.aliyuncs.com/csw/游戏资讯日报/2026-06-04/02-选题/游戏资讯日报_选题研究员_20260604_v1.zip
> ```
>
> 下文 header 里的 `上游来源` 均按此写全。把示例域名 `csw-oss.oss-cn-hangzhou.aliyuncs.com` 换成你实际的 OSS 域名即可。

## 5. 溯源：index.md 元信息头

每个 `index.md` 顶部放一段 YAML 元信息头，作为「谁做的 + 完整链路」的权威记录：

```yaml
---
任务：游戏资讯日报
类型：文案成品 / 派工单 / 封面 / 文章 / 成品 / 发布物
交付 Agent：选题研究员
阶段：02-选题
版本：v1
时间：2026-06-04 13:40
上游来源：主编 · https://csw-oss.oss-cn-hangzhou.aliyuncs.com/csw/游戏资讯日报/2026-06-04/02-选题/游戏资讯日报_主编_20260604_v1.zip
状态：待主编审核 / 待用户审核 / 已退回（附说明）/ 已通过
自检：（按各阶段自检清单填）
---
```

`上游来源` 写「来源方 · 完整 https 链接」；多个上游就逐条列。文件名负责「一眼是谁做的」，header 负责「权威记录 + 完整上游链路」，两层并存、不互相替代。

## 6. 单文件归属

多数情况类型即归属：
- `index.md` / 正文文章 = **深度内容作者**
- `images/` 封面图 = **视觉设计师**
- `source/` 资讯原文、链接清单、截图 = **游戏情报员**
- 标题 / 摘要（写在 header 或正文）= **选题研究员**
- `html/` 网页文件 = **发布员**
- 派工单 / 意见 = **主编**（写在各派工包的 `index.md` 里）

模糊的（如主编要求补的图）在 `index.md` 文件清单里注明产出方。

## 7. 审核与退回（双闸 + 主编中枢）

**每一阶段都走两道闸，顺序固定：**

1. **主编审** — 对照本阶段验收标准（细则见《游戏资讯日报 · 验收标准》）。
   - 不过：主编退回该 Agent，**告知要改的方向和具体位置**（主编不代改）→ Agent 改后**升版本**重打 zip、原版留底 → 再交主编审。
   - 过：主编转交用户审。
2. **用户审** — 由主编代为转交。
   - 不过：主编退回该 Agent 重做（升版本），告知方向和位置。
   - 过：进入下一步。

**用户通过后，主编做一次「重新打包 + 写意见和下一步安排」**，产出**派工单**（责任方=主编，类型=派工单），把上游交付物的**完整 https 链接**、主编意见、下一步安排都写进派工单的 `index.md`，再派给下一个 Agent。

**状态流转**（写在 `index.md` 的 `状态` 字段）：
`待主编审核` →（主编过）`待用户审核` →（用户过）`已通过` ；任一不过 → `已退回（附说明）`。
