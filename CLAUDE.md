# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 仓库性质

本仓库定义微信公众号「每日花生新闻」之「游戏资讯日报」任务的多 Agent 协作流程：一套中文 Markdown 规范文档（源头），加上由它们转写生成的 6 个可安装角色 skill（派生物）。无构建/测试命令。

## 文档结构与权威关系

- `游戏资讯日报_交付与流转规范.md` — **总规范（权威文档）**：定义交付形态、命名、OSS 路径、溯源 header、审核流转等所有通用机制。
- `游戏资讯日报_验收标准.md` — **主编审核用的验收细则**：通用合规项 + 六阶段分组核对清单（内容质量/规范合规）。条目全部提炼自各角色手册的规格与自检，不新增要求；主编手册第 6 节是它的速查摘要。
- `作业手册_*.md` — 6 份角色作业手册（主编、游戏情报员、选题研究员、视觉设计师、深度内容作者、发布员），各自描述该角色的输入、产出、自检清单。每份手册结尾都引用总规范。
- `编辑部通讯录.md` — 飞书群「编辑部」各 Agent 的 Open ID（群消息 mention 用）。
- `html-guide.md` / `templates/` — 微信公众号 HTML 规范与文章样式模板（发布员专用）。
- `skills/` — 6 个**自包含**的角色 skill（`lemon-chief-editor` / `lemon-game-intel` / `lemon-topic-researcher` / `lemon-cover-designer` / `lemon-content-writer` / `lemon-publisher`），由对应作业手册忠实转写；共享文件（通讯录、总规范、验收标准、html-guide、templates）以**副本**形式复制在各 skill 的 `references/`、`assets/` 内。
- `install/` — 6 份发给各 Agent 的自助安装文档（git clone + 复制到 `~/.claude/skills/`）；README 中的对照表链接即指向这里。

**修改时的一致性要求**：通用机制（命名规则、OSS 路径、header 字段、审核流程、版本规则）以总规范为准；改动这类内容必须同步更新所有引用它的作业手册。角色专属细节（如封面规格、采集节奏）只存在于对应手册中；但存在如下**镜像关系，改一处必须同步其余**：

1. 角色手册的规格/自检条目 ↔ `游戏资讯日报_验收标准.md` 对应阶段清单 ↔ 主编手册第 6 节速查表
2. 角色手册 ↔ `skills/<对应skill>/SKILL.md` 正文
3. 仓库根的共享文件（通讯录 / 总规范 / 验收标准 / html-guide / templates）↔ 各 skill 内的同名副本

Skill 通过 GitHub 分发（agent 收到 `install/` 文档链接后 git clone 本仓库安装），因此 skill 相关改动**必须 push 到 GitHub 才生效**，并需通知各 Agent 重新执行安装文档。

## 核心架构：六阶段流水线

```
用户 → 主编(启动派工单) → 01-情报(游戏情报员) → 02-选题(选题研究员)
        ↓ (02 通过后，用户把封面素材交主编，主编同时派两路)
   ┌── 03-视觉(视觉设计师)    ← 两路并行
   └── 04-内容(深度内容作者)
        ↓ (两路都过用户审后)
   05-成品(主编合并打包) → 06-发布(发布员) → 完成
```

**主编是中枢且唯一对接用户的人**：只审核、退回、转交、重新打包派工，**绝不亲自编辑内容**。每阶段走两道闸：主编审 → 用户审（主编代为转交）。任一不过即退回责任 Agent 重做并升版本。

## 关键机制（跨文件不变量）

- **交付形态**：每份交付物 = 文件夹（`index.md` 入口 + 附件子目录如 `images/`、`source/`、`html/`）→ 打包同名 zip → 用 `aliyun-oss-delivery` skill 上传 OSS → Agent 之间只流转**完整 https 链接**（不能只写路径）。
- **命名**：`游戏资讯日报_责任方_日期_版本.zip`。主编派工单责任方=主编；多方合成责任方=阶段名（如「成品」）。
- **OSS 路径**：`csw/游戏资讯日报/{日期}/{阶段序号-阶段}/{zip}`，阶段序号固定为 01-情报 / 02-选题 / 03-视觉 / 04-内容 / 05-成品 / 06-发布。派工单归到**目标阶段**路径下。
- **溯源**：每个 `index.md` 顶部必有 YAML 元信息头（任务/类型/交付 Agent/阶段/版本/时间/上游来源/状态/自检）。`上游来源` 写「来源方 · 完整 https 链接」。
- **版本与留底**：退回重做升版本（v1/v2/v3…），每版各打 zip 各自留底，原版不删。
- **状态流转**：`待主编审核` → `待用户审核` → `已通过`；任一不过 → `已退回（附说明）`。
- **群消息**：编辑部群内发新消息（不用回复形式），`lark-cli messages-send --msg-type text` + `<at user_id="ou_xxx">名字</at>` 真实 mention（ID 见通讯录），禁止纯文本 @。

## 角色绑定的工具与硬性约束

修改手册或 skill 时不要破坏这些约定：

- **游戏情报员**：先读派工单四要素（时间范围/重点对象/平台指令/指定网站）；只用 `opencli reddit` / `opencli twitter`（指定网站用 `opencli browser game-news`）采集，每步插入随机 sleep（防风控）；内容分「官方情报（完整内容）/ 社区玩家讨论（热门内容）」；图片须经 `vision_analyze` 筛选，优先 1080p、最低 720p，每条资讯合格图 ≥6 张。
- **选题研究员**：文章主标题 ≤64 字符（发布员用作公众号标题）；封面标题最多两行、每行 ≤5 字（视觉设计师放到封面）；摘要 ≤120 字符。
- **视觉设计师**：封面必须用 `generate-images` skill 生成；比例 2.35:1，优先 1920×817 px；左侧角色区 / 右侧 1:1 标题安全区版式，文字不得越界到左侧。
- **深度内容作者**：逐条整理，新游戏发布条目含发售时间/平台/港区价格；**正文不含文章主标题和摘要**（发布员单独填入平台字段）；末尾页脚两行——`数据来源：`按日替换，「整理：Lemon，「每日花生新闻」……」逐字固定不改。
- **发布员**：用 `md2wechat` skill 转 HTML，按当期类型套 `templates/` 模板（日常/任天堂=红 `#e60012`、索尼=蓝 `#011def`、Xbox=绿 `#107C10`）；图片排版按 1/2/3/4 张规则；对照 `html-guide.md` 验证（全部内联 style）；插入两个广告组件（前置正文各 >300 字、位于模块末段 `<p>` 后 `</section>` 前、代码属性原样）；用 `mp-helper` skill 上传图片入素材库并替换链接、上传草稿箱；作者固定为 `CCS_Lemon`；上传后**先通知主编**，不直接通知用户。
