# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 仓库性质

这是一个**纯文档仓库**（无代码、无构建/测试命令），定义微信公众号「游戏资讯日报」的多 Agent 协作流程规范。所有文件均为中文 Markdown。

## 文档结构与权威关系

- `游戏资讯日报_交付与流转规范.md` — **总规范（权威文档）**：定义交付形态、命名、OSS 路径、溯源 header、审核流转等所有通用机制。
- `游戏资讯日报_验收标准.md` — **主编审核用的验收细则**：通用合规项 + 六阶段分组核对清单（内容质量/规范合规）。条目全部提炼自各角色手册的规格与自检，不新增要求；主编手册第 6 节是它的速查摘要。
- `作业手册_*.md` — 6 份角色作业手册（主编、游戏情报员、选题研究员、视觉设计师、深度内容作者、发布员），各自描述该角色的输入、产出、自检清单。每份手册结尾都引用总规范。

**修改时的一致性要求**：通用机制（命名规则、OSS 路径、header 字段、审核流程、版本规则）以总规范为准；改动这类内容必须同步更新所有引用它的作业手册。角色专属细节（如封面规格、采集节奏）只存在于对应手册中；但改动任何角色手册的**规格或自检条目**时，必须同步更新 `游戏资讯日报_验收标准.md` 对应阶段的清单及主编手册第 6 节速查表（三处镜像关系）。

## 核心架构：六阶段流水线

```
01-情报(游戏情报员) → 02-选题(选题研究员)
        ↓ (02 通过后，主编同时派两路)
   ┌── 03-视觉(视觉设计师)    ← 两路并行
   └── 04-内容(深度内容作者)
        ↓ (两路都过用户审后)
   05-成品(主编合并打包) → 06-发布(发布员) → 完成
```

**主编是中枢且唯一对接用户的人**：只审核、退回、转交、重新打包派工，**绝不亲自编辑内容**。每阶段走两道闸：主编审 → 用户审（主编代为转交）。任一不过即退回责任 Agent 重做并升版本。

## 关键机制（跨文件不变量）

- **交付形态**：每份交付物 = 文件夹（`index.md` 入口 + 附件子目录如 `images/`、`source/`、`html/`）→ 打包同名 zip → 上传 OSS → Agent 之间只流转**完整 https 链接**（不能只写路径）。
- **命名**：`游戏资讯日报_责任方_日期_版本.zip`。主编派工单责任方=主编；多方合成责任方=阶段名（如「成品」）。
- **OSS 路径**：`csw/游戏资讯日报/{日期}/{阶段序号-阶段}/{zip}`，阶段序号固定为 01-情报 / 02-选题 / 03-视觉 / 04-内容 / 05-成品 / 06-发布。派工单归到**目标阶段**路径下。
- **溯源**：每个 `index.md` 顶部必有 YAML 元信息头（任务/类型/交付 Agent/阶段/版本/时间/上游来源/状态/自检）。`上游来源` 写「来源方 · 完整 https 链接」。
- **版本与留底**：退回重做升版本（v1/v2/v3…），每版各打 zip 各自留底，原版不删。
- **状态流转**：`待主编审核` → `待用户审核` → `已通过`；任一不过 → `已退回（附说明）`。

## 角色绑定的工具与硬性约束

修改手册时不要破坏这些约定：

- **游戏情报员**：只用 `opencli reddit` / `opencli twitter` 采集，每步之间必须插入随机 sleep 间隔（防风控）；内容分「官方情报（完整内容）/ 社区玩家讨论（热门内容）」两类。
- **选题研究员**：产出两个标题——文章主标题 ≤64 字符（发布员用作公众号标题）；封面标题最多两行、每行 ≤5 字（视觉设计师放到封面上）。
- **视觉设计师**：封面必须用 `generate-images` skill 生成；比例 2.35:1，优先 1920×817 px；左侧角色区 / 右侧 1:1 标题安全区版式，文字不得越界到左侧。
- **发布员**：用 `mp-helper` skill 把 HTML 内图片上传公众号素材库并替换链接，再上传草稿箱；作者固定为 `CCS_Lemon`；上传后**先通知主编**，不直接通知用户。
